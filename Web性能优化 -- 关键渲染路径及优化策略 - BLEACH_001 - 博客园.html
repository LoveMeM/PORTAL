Web性能优化 -- 关键渲染路径及优化策略

一、前言

二、概述

　　总体来说，当浏览器从服务器接收到HTML、CSS和JavaScript 之后，中间会经过几个步骤来将其渲染到浏览器上。

  1. 处理HTML标记并构建DOM树
  2. 处理CSS标记并构建CSSOM树
  3. 将DOM和CSSOM合并形成渲染树
  4. 根据渲染树来布局(Layout)，以计算每个节点的几何信息
  5. 将各个节点绘制到屏幕上

　　

 

　　Note：如果DOM或者CSSOM被修改，浏览器将会重新执行一遍上面的步骤，以确定哪些像素或者说元素需要重新被渲染（重绘和重排将会详细说明）。

三、关键渲染路径　

　　什么叫关键渲染路径？关键渲染路径就是描述浏览器从收到 HTML、CSS 和 JavaScript 字节开始，到如何使用HTML、CSS 和 JavaScript 在屏幕上渲染像素的中间过程。如果我们能优化这些中间步骤，即缩短上述1~5步的总体时间，更快的将元素渲染到页面上，让用户感觉到页面被更快的加载出来，由此提高用户体验。

　　首先我们先了解一下浏览器是如何处理从1~5的中间流程，从而更好的有针对性的来进行优化。

　　构建对象模型

　　　　浏览器渲染页面前先需要构建DOM和CSSOM树，

　　　　构建DOM树（文档对象模型）

 1 <html>
 2   <head>
 3     <meta name="viewport" content="width=device-width,initial-scale=1">
 4     <link href="style.css" rel="stylesheet">
 5     <title>Critical Path</title>
 6   </head>
 7   <body>
 8     <p>Hello <span>web performance</span> students!</p>
 9     <div><img src="awesome-photo.jpg"></div>
10   </body>
11 </html>

　　上述HTML代码是一个很简单的例子，一个包含一些文本和一副图片的普通HTML页面，那么浏览器会如何去构建此页面的DOM树？

　　

　　浏览器每一次处理HTML标记时，都会完成以上的所有步骤，将字节转化为字符串，然后确定Token，再将Token转化为节点对象，然后构建DOM树。这整个流程可能需要一些时间才能完成，当有大量的HTML需要处理的时候更是如此。 

　　CSS对象模型（CSSOM）

  　 在浏览器构建我们简单页面的DOM时，在文档的Head部分遇到link标记，该标记引用了一个外部CSS样式表。此时浏览器知道需要依靠此外部样式表来渲染页面， 它会立即发出对该资源的请求，并返回一下内容。

1 body { font-size: 16px }
2 p { font-weight: bold }
3 span { color: red }
4 p span { display: none }
5 img { float: right }

　　和构建DOM树类似，我们需要将收到的CSS转化为浏览器可以理解和处理的东西，此时会重复HTML过程，只不过最后生成的是CSSOM树

　　

 

还请注意，以上树并非完整的 CSSOM 树，它只显示了我们决定在样式表中替换的样式。每个浏览器都提供一组默认样式（也称为“User Agent 样式”），即我们不提供任何自定义样式时所看到的样式，我们的样式只是替换这些默认样式。 

CSSOM 和 DOM 是独立的数据结构！都是独立的对象，分别网罗文档不同方面的信息：一个描述内容，另一个则是描述需要对文档应用的样式规则。

　　渲染树(Render Tree)构建、布局(Layout)及绘制

　　CSSOM树和DOM树合并成渲染树，然后用于计算每个可见元素的布局，并输出绘制流程，然后将像素渲染到屏幕上

  • DOM 树与 CSSOM 树合并后形成渲染树
  • 渲染树只包含渲染网页所需的节点
  • 布局计算每个对象的精确位置和大小
  • 最后一步是绘制，使用最终渲染树将像素渲染到屏幕上。

　　

　　

　　为构建渲染树，浏览器大体上完成了下列工作：

  1. 从 DOM 树的根节点开始遍历每个可见节点。
    • 某些节点不可见（例如脚本标记、元标记等），因为它们不会体现在渲染输出中，所以会被忽略。
    • 某些节点通过 CSS 隐藏，因此在渲染树中也会被忽略，例如，上例中的 span 节点不会出现在渲染树中，因为有一个显式规则在该节点上设置了“display: none”属性。
  2. 对于每个可见节点，为其找到适配的 CSSOM 规则并应用它们。
  3. 发射可见节点，连同其内容和计算的样式。

　　最终输出的渲染同时包含了屏幕上的所有可见内容及其样式信息。有了渲染树，我们就可以进入“布局”阶段。

 　　Layout

　　到目前为止，我们计算了哪些节点应该是可见的以及它们的计算样式，但我们尚未计算它们在视窗内的确切位置和大小---这就是“布局”阶段，也称为“自动重排”。

　　为弄清每个对象在网页上的确切大小和位置，浏览器从渲染树的根节点开始进行遍历。让我们考虑下面这样一个简单的实例：

 1 <html>
 2   <head>
 3     <meta name="viewport" content="width=device-width,initial-scale=1">
 4     <title>Critial Path: Hello world!</title>
 5   </head>
 6   <body>
 7     <div style="width: 50%">
 8       <div style="width: 50%">Hello world!</div>
 9     </div>
10   </body>
11 </html>

　　布局流程的输出是一个“盒模型”，它会精确地捕获每个元素在视口内的确切位置和尺寸：所有相对测量值都转换为屏幕上的绝对像素。最后我们知道了那些节点可见、它们的计算样式以及几何信息，此时就可以进行最后一个阶段，将渲染树中的每个节点转化为屏幕中的实际像素，这一步称为“绘制”。

　　阻塞渲染的资源（CSS，JavaScript）

　　对于HTML和CSS来说，毫无疑问，因为渲染树的构建需要DOM数和CSSOM树，所以CSS和HTML肯定是阻塞资源。但是对于JavaScript来说，事情就相对复杂的多。

　　既然CSS是阻塞渲染的资源，这就意味着在CSSOM构建完成之前，浏览器不会去渲染任何已处理的内容。要尽早、尽快地把CSS下载到客户端以优化首次渲染的时间，所以会将link标签放在head里面，让浏览器尽量早的识别到link请求，从而发起css请求。

 　　但是对于JavaScript来说，当HTML解析的过程中遇到<Script>标签时，会暂停DOM的构建，将控制权交给JavaScript引擎，等JavaScript引擎运行完毕之后再从中断的地方重新构建DOM。大家应该都知道，脚本在那里插入就在那里执行。那么对于内联的JavaScript来说，就会阻塞页面的首次渲染。

　　但是如果JavaScript是外部资源呢？此时浏览器渲染会阻塞直到外部请求返回并执行了JavaScript之后再继续构建DOM。但是JavaScript有一个不同的逻辑，就是JavaScript中可以操作DOM和CSS，那么如果当HTML文档中即存在link标签又存在script标签的时候，此时JavaScript会等待CSSOM构建完毕之后再去执行JavaScript脚本 。

　　举例：

　　Demo1

<html>
  <head>
    <meta name="viewport" content="width=device-width,initial-scale=1">
    <title>Critical Path: No Style</title>
  </head>
  <body>
    <p>Hello <span>web performance</span> students!</p>
    <div><img src="awesome-photo.jpg"></div>
  </body>
</html>

　　Demo2：

<html>
  <head>
    <meta name="viewport" content="width=device-width,initial-scale=1">
    <link href="style.css" rel="stylesheet">
  </head>
  <body>
    <p>Hello <span>web performance</span> students!</p>
    <div><img src="awesome-photo.jpg"></div>
  </body>
</html>

　　Demo3：

<html>
  <head>
    <meta name="viewport" content="width=device-width,initial-scale=1">
    <link href="style.css" rel="stylesheet">
  </head>
  <body>
    <p>Hello <span>web performance</span> students!</p>
    <div><img src="awesome-photo.jpg"></div>
    <script src="app.js"></script>
  </body>
</html>

五、优化策略

　　　

  1. 减少关键资源大小

　　　　对于所有的资源（HTML，JavaScript，CSS，Image等），你都应该用上三大绝招：Minify，Compression和Cache，这里不过多的赘述里面的细节。

　　　　这一点对于HTML来说，非常关键，HTML作为渲染的关键资源，消除或者延迟加载肯定不太可能(这里指的是非局部渲染的关键HTML)，能够做到是消除无用代码（比如：注释）和最小化代码（Minify）以及动态局部渲染等。

　　2.延迟JavaScript非阻塞资源

　　　　async，完全的异步操作，HTML解析遇到该标签后，发出网络请求，但不阻止HTML解析和其后面的渲染操作，当JavaScript请求返回后立刻执行，且不等待HTML解析或其他操作的完成。所以，如果脚本中有DOM操作，就并不适合。比较适合的场景是Google Analytics。

　　　　defer，HTML的解析和对JavaScript资源的网络请求是并行的，但它会等待HTML解析完成之后，才执行脚本

　　3.尽早的加载CSS和尽量少用@Import标签

　　4.内联样式表，减少http请求。
